See your tier Quickstart

The headline feature is the policy that decides how little to stream.

Streaming transformer weights layer-by-layer is the fallback that makes otherwise-impossible models merely slow. It is never the fastest path when memory exists. streamllm auto-selects the least-streaming tier and logs the numbers that forced it.

TierWhen it is chosenWhat happensHonest expectation
full Weights + KV at max context + activations fit the accelerator No streaming. Plain full load. Use vLLM / TGI / llama.cpp. They will be faster. streamllm says so, then runs.
gpu_ram Does not fit VRAM, fits host RAM, 2+ layers resident on a discrete GPU Resident set on GPU; stream the rest from pinned host RAM, LRU + async prefetch Slower than Tier 0, usually compute-bound if the bus keeps up
ram Fits host RAM, little or no usable GPU, or unified memory All decoder weights in RAM; stream into compute. Unified memory is residency, not a real copy Workable. On unified memory, close to native.
disk Does not fit RAM Stream layers from mmap'd on-disk shards I/O-bound by physics. The benchmark reports the disk read ceiling you actually hit.

Honest by construction

Tier 0 prints that a dedicated engine is faster. Tier 3 prints the effective disk read ceiling it actually hit. The benchmark reports median and p90, time-to-first-token, and an I/O-vs-compute verdict. No marketing math.

Correctness-first streaming

Weights stream via forward hooks while the model's own attention, RoPE and KV-cache math run untouched. A verify-against-full-load test confirms streamed logits match a plain full load for prefill and multi-step decode.

The memory model gets KV right

KV cache uses num_key_value_heads, not num_attention_heads. A 0.9 headroom factor guards against "fits at load, OOMs as KV grows". If a tier OOMs for real, streamllm demotes one tier with a loud warning instead of crashing.

Disk shards that amortize

One safetensors file per layer with a hashed manifest. Sharding is resumable and verified. Quantized shards (int8 / int4) read fewer bytes per layer, and batching makes one layer read serve many sequences.

Limitations, first

The disk tier is I/O-bound by physics.

Reading every layer per token is disk-bandwidth-limited. Quantized shards, larger batches and prefetch overlap reduce the wall. None of them remove it. streamllm says this in logs and reports rather than implying otherwise. If your model fits, the right answer is a dedicated engine, and the library will tell you.

Quickstart

Install, point at any Llama- or Qwen-family model, and stream tokens. Plan a model you cannot even run with streamllm describe, no weights loaded.
